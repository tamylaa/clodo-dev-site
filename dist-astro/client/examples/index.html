üéØ Common Implementation Patterns

Authentication Flow

// User logs in with credentials
POST /auth/login
  -> Validate credentials against D1 database
  -> Generate JWT token
  -> Return token + refresh token to client

// Client includes token in requests
GET /api/resource
  Authorization: Bearer <token>
  -> Verify JWT signature
  -> Extract user ID from claims
  -> Scope all data operations to user
  -> Return scoped resource

Multi-Tenant Data Isolation

// Request context includes tenant information
const ctx = {
  tenantId: "acme-corp",
  userId: "user-123",
  permissions: ["read", "write"]
}

// All database queries automatically scoped
const orders = await db.table('orders')
  .where('tenant_id', '=', ctx.tenantId)
  .where('user_id', '=', ctx.userId)
  .select();

// If someone tries to bypass tenant_id, query fails
// (Clodo enforces this automatically)

Webhook Handling

// Receive webhook from external service
POST /webhooks/stripe
  -> Verify webhook signature with secret
  -> Parse event data
  -> Queue processing job
  -> Respond 200 OK immediately

// Process webhook asynchronously
  -> Look up customer in database
  -> Update subscription status
  -> Send confirmation email
  -> Retry on failure with exponential backoff

Rate Limiting

// Clodo tracks request counts per user
GET /api/expensive-operation
  -> Check rate limit (100 requests/hour)
  -> Decrement counter
  -> If exceeded: 429 Too Many Requests
  -> Include Retry-After header

// Built-in sliding window algorithm
  -> Prevents burst attacks
  -> Fair distribution across time period
  -> Works across all edge locations
